# Production Hybrid Search Benchmark

Follow-up of [#31](https://github.com/dignite-projects/dignite-paperbase/issues/31).
Validates the Slice 7 hybrid search claim with real infrastructure and a desensitized corpus.

## Prerequisites

| Requirement | Notes |
|-------------|-------|
| PostgreSQL ≥ 15 + pgvector + all migrations applied | `dotnet run --project host/src/Dignite.Paperbase.DbMigrator` |
| Embedding model access | Azure OpenAI or Ollama — needed once for dataset generation |
| .NET 10 SDK | Same SDK used to build the main solution |

## Step 1 — Prepare the gold dataset

The benchmark reads from `docs/benchmarks/rag-gold-dataset.json` (this file is **not** committed to the repo — it contains desensitized real data that must be prepared offline).

### Dataset format

See `rag-gold-dataset-sample.json` for the schema. The file contains:

- **`chunks`** — desensitized document chunks. Each chunk needs:
  - `id`: stable UUID (used as ground truth key)
  - `text`: desensitized chunk text (no PII, replace names/IDs with synthetic ones)
  - `documentTypeCode`: e.g. `"contract"`, `"invoice"`, `"certificate"`
  - `embeddingBase64`: Base64-encoded `float32[]` (little-endian), generated by running the real embedding model on `text`
- **`queries`** — labeled queries, each with:
  - `id`: stable string
  - `text`: the query as a user would type it
  - `category`: `"precise-text"` or `"semantic"`
  - `expectedChunkIds`: list of chunk UUIDs that are correct answers
  - `embeddingBase64`: same encoding as above

### Corpus size target

| Category | Chunks | Queries |
|----------|--------|---------|
| Contracts | 50–100 | 25 precise-text |
| Invoices | 30–60 | 15 precise-text |
| Certificates | 20–40 | 10 precise-text |
| Prose / policies | 10–20 | 15 semantic (multi-chunk OK) |
| Clause / paraphrase | — | 35 semantic |

Minimum viable corpus for meaningful statistics: 50 queries per category.

### Generating embeddings

Use the same model and endpoint as the production deployment (configured in `appsettings.json` under `PaperbaseRag:EmbeddingDimension` — currently 1536, meaning `text-embedding-3-small`).

Example script (Python, using `openai` SDK):

```python
import openai, json, struct, base64, uuid

client = openai.AzureOpenAI(...)  # configure your endpoint

def embed(text: str) -> str:
    resp = client.embeddings.create(input=text, model="text-embedding-3-small")
    floats = resp.data[0].embedding
    raw = struct.pack(f"{len(floats)}f", *floats)
    return base64.b64encode(raw).decode()

# Load your desensitized text, call embed(), store result
```

The `struct.pack(f"{N}f", ...)` produces little-endian float32 bytes, which is what the benchmark reader expects.

### Privacy checklist before saving the dataset

- [ ] No real names (replace with 山田A, 鈴木B, … or synthetic names)
- [ ] No real contract/invoice numbers (replace with CNT-XXXX, INV-XXXX patterns using different sequences from production)
- [ ] No real dates, amounts, or addresses that could identify specific transactions
- [ ] No real employee/student IDs
- [ ] Store the file locally — do **not** commit to the git repo

## Step 3 — Run the benchmark

```bash
dotnet test core/ \
  --filter "Category=Production" \
  -- RunConfiguration.TestSessionTimeout=300000
```

The test:
1. Inserts the benchmark chunks (with pre-computed vectors) into the real PostgreSQL
2. Runs every query in both `Vector` and `Hybrid` modes
3. Computes recall@1, recall@5, MRR per category
4. Appends results to `docs/design/rag-hybrid-benchmark-2026Q2.md`
5. Asserts acceptance thresholds (precise-text lift ≥ 0.03, semantic regression ≤ 0.03)
6. Deletes all seeded rows in a `finally` block (safe to Ctrl+C — cleanup runs on test teardown)

## Query design rules for the gold dataset

### Precise-text queries must be pure identifiers

Write precise-text queries as the bare ID the user would search for, **not** as a natural-language sentence with an embedded ID:

```
✅  CNT-2025-0001
✅  契約番号 CNT-2025-0001 委託料        ← space-separated tokens
❌  CNT-2025-0001の委託料はいくらですか   ← "の委託料はいくらですか" is a huge
                                          single token that won't match any chunk
```

`plainto_tsquery('simple', ...)` requires **all** parsed tokens to appear in the chunk's `SearchVector`. Japanese text with no spaces becomes one giant token. If the query contains Japanese after the ASCII ID, the `@@` match fails silently and the keyword path returns nothing — making Hybrid identical to Vector.

### ASCII identifiers need a space before them in chunk text

The `simple` regconfig tokenizes `契約番号CNT-2025-0001` (no space) as a **single** token `契約番号cnt-2025-0001`, not as three tokens `'cnt' & '-2025' & '-0001'`. This breaks keyword matching.

Ensure the chunking pipeline (or your hand-written corpus) puts a space between Japanese text and the ASCII identifier:

```
✅  契約番号 CNT-2025-0001 。
❌  契約番号CNT-2025-0001。
```

### Schema sync: benchmark SQL must match the current entity schema

`ProductionHybridSearchBenchmark.cs` uses raw ADO.NET INSERTs that name columns explicitly. When the `Document` or `DocumentChunk` entity changes (columns renamed, added, or removed), the benchmark SQL **does not break at compile time** — it only fails at runtime with a Postgres `42703` error.

If you rename a column in the domain entity, update the corresponding column list in `SeedDocumentsAsync` / `SeedChunksAsync` in the benchmark file as well.

## Troubleshooting

| Symptom | Likely cause |
|---------|-------------|
| Test skips immediately | `PAPERBASE_BENCH_PGCONN` not set |
| `relation "PaperbaseDocumentChunks" does not exist` | Migrations not applied — run DbMigrator first |
| `column "SearchVector" of relation ... does not exist` | Migration `20260428004038_Slice7_AddDocumentChunkSearchVector` not applied |
| `column "XYZ" of relation "PaperbaseDocuments" does not exist` | Domain entity was changed — update `SeedDocumentsAsync` column list to match |
| Hybrid == Vector (identical scores) | Keyword path returning no rows — check query design (pure-ID queries) and chunk text spacing (space before ASCII IDs) |
| Vector search returns no results | Embeddings in dataset may be all-zero placeholders — regenerate with real model |
| Unique constraint on `OriginalFileBlobName` | Previous run's cleanup failed; delete stale rows with `DELETE FROM "PaperbaseDocuments" WHERE "OriginalFileBlobName" LIKE 'bench:%'` |

## Re-running

The benchmark is fully idempotent — each run seeds fresh rows with new UUIDs and deletes them on completion. Re-run any time embeddings or corpus change:

```bash
dotnet test core/ --filter "Category=Production"
```
